## 1. 課題・ゴール

### 1.1 課題
- 半日有休を取得した場合に、**日報入力データと勤怠（Jobcan）データで齟齬** が出るケースがある。
- 原因はスタッフごとに「休暇タイプ」が異なり「半日」の時間が変動すること。さらに昼休憩の自動計算など複数要素が絡み、**正しい時間を自動算出するロジックが組めない**。

### 1.2 やること（確定）
- **UI側で各スタッフが手動訂正できれば十分** とする。
- 「有休」タスクがある日に限り、有休タスクの **時間 / 時間帯 / `task_total_minutes`** に不整合があれば、**入力値で上書きできる** 機能で完結させる。
- Jobcanから受けた値は **参考値としてUIに提示**。間違いがあればユーザー自身が修正し確定する。

### 1.3 やらないこと（確定）
- **JobcanAPIで適切なデータを得る根本対策は諦める。再検討しない。**
- 現状「中途半端に機能している休暇API取得部分」は **変更も削除もしない**（参考値取得として温存）。
- 半休時間の自動正確計算（スタッフ種別・休暇タイプ・昼休憩を加味した算出）は目指さない。

## 2. 現状の仕組み（コードから確認した事実）

> 行番号は 2026-06-22 時点。改修日までにズレる可能性があるため目安。

### 2.1 全体構成（README準拠）
- バックエンド: Flask（`main.py`） / Cloud Run、Firestore。
- 入力UI（スマホ）: `liff-app.js`（工務・ネット両対応、`isReportNetPage` で分岐）。
- 管理画面（PC）: `admin.html` / `admin_net.html` + `js/admin-app.js`、代理入力 `_manager_proxy_report.html` / `_manager_proxy_report_net.html`。
- API: `routes/api.py`。休暇関連ロジックの多くは `app_core/utils.py`。

### 2.2 日報タスクのデータ形状（重要な差分）
`liff-app.js` の送信処理（`1234–1277`）で **工務とネットでタスク形状が異なる**:

| 区分 | エンドポイント | タスクの主なフィールド | 時間の決まり方 |
|------|----------------|------------------------|----------------|
| 工務 | `POST /api/reports` | `categoryA_id/label`, `categoryB_id/label`, **`time`(分)** | `time` を **分で直接入力**（free input） |
| ネット | `POST /api/reports_net` | 上記 + **`startTime`/`endTime`**, `comment` | `startTime`/`endTime`(15分丸め) から **分を導出** |

- 送信ボディ共通: `date`, `taskTotalMinutes`(=全タスク`time`合計 `allocatedMinutes`), `jobcanWorkMinutes`, `tasks`。
- 工務は **有休タスクの`time`を画面でそのまま編集可能**（自由入力）。ネットは **時間帯から導出** されるため、有休タスクの訂正導線が工務と異なる。← **改修ボリュームの主因**

### 2.3 入力画面の既存「差異」挙動（`liff-app.js`）
- `allocatedMinutes`（タスク合計） vs `totalWorkTime`（勤務時間）の差を計算し、`remainingMinutes !== 0` のとき送信時に確認ダイアログ（`1223–1224`「勤務時間と入力した工数に差異があります。このまま送信しますか？」）。
- カテゴリ欄は `readonly`（`292–304`）。**時間欄は編集可**。
- `1552–1569`: 有休情報を取得して `calendarStatuses[date].paid_holiday = 'full'|'half'` にマージ。カレンダー表示用。

### 2.4 有休のJobcan同期（`routes/api.py`）
- `POST /api/sync-paid-holidays`（`5564–`）: 月度の有休等をJobcanから取得し日報へ反映。
- `GET /api/jobcan/paid-holidays`（`5333–`）: 参考値取得。`liff-app.js 3107` の「更新」ボタンから呼ばれる。
- `register_paid_holiday_work_report`（`6010–6158`）が有休タスクを生成/更新:
  - 有休タスク固定値: `categoryA_id="A00"`, `label="有休"`, `categoryB_id="e_000000"`, `label="000000"`, `detail="有休自動適用"`。
  - ネット形式時のみ `startTime`/`endTime`/`comment` を付与（`6055–6058`）。
  - **既存ドキュメントに有休らしきタスクが1件でもあれば、追加も置換もしない**（`6071–6080`、`_task_looks_like_paid_leave` 判定）。
    → **手動訂正後に再同期しても上書きされない**（= やりたいことと整合。重要な既存挙動）。
  - 追加時は `task_total_minutes` を全タスク再合計（`6086–6096`）。

### 2.5 半休の分数決定（`app_core/utils.py`）
- 工務: `resolve_paid_leave_minutes_engineering`（`1193–`）。`holiday_types.minutes` 優先、なければ **full=480 / half=240 にフォールバック**（`1204`）。
- 同期用: `resolve_paid_leave_for_sync`（`1212–`）/ `..._by_work_kind`（`1249–`）が minutes と holiday の開始/終了時刻を返す。
- ネット半休で開始/終了が取れない場合のフォールバック開始時刻推定: `api.py 5504–`。

### 2.6 齟齬が生まれるメカニズム（整理）
1. 半休の実時間がスタッフ種別/休暇タイプ/昼休憩で変動するのに、同期は `holiday_types.minutes` か **240分の固定フォールバック** で有休タスクを作る。
2. その結果、有休タスク`time`（および`task_total_minutes`）が実態とズレ、`jobcan_work_minutes` との突合（予実）で差異として現れる。
3. ネットは `startTime`/`endTime` 由来のため、時間帯のズレも齟齬要因になる。

## 3. 解決アプローチ（複数案を列挙）

> 共通方針: 「有休タスクのある日」を検出 → 参考値（Jobcan）と入力値を提示 → ユーザーが上書き確定。
> 再同期で上書きされない既存挙動（2.4）を前提にできるのが追い風。

### 案A: 既存の時間入力欄を活かす最小改修（推奨の出発点）
- **工務**: 有休タスクの`time`欄は既に編集可能。**「有休あり日」を検知して参考値バッジ/警告を表示** + 確定時に `task_total_minutes` を正しく再計算するだけ。
- **ネット**: 有休タスクの `startTime`/`endTime` を編集→分を導出する既存導線を有休タスクにも適用し、参考値を併記。
- メリット: 改修面が小さい。新規UIコンポーネントほぼ不要。既存フォームの延長。
- デメリット: 「不整合の検知・可視化」をどこまで作るかで工数が動く。ネットの有休タスク編集導線が現状どこまで効くか要確認。
- 向くケース: まず工務スマホでMVPを通したい場合。

### 案B: 「齟齬バナー＋ワンクリック上書き」UI（体験重視）
- 有休あり日で「参考値 N分 / 入力値 M分 が不一致」をバナー表示し、「参考値で上書き」「入力値を維持」を選べる。
- メリット: ユーザーが齟齬を見落としにくい。訂正操作が明快。
- デメリット: 検知ロジック＋UIコンポーネントを4面ぶん作る必要があり工数増。
- 向くケース: 運用者がミスを減らしたい/問い合わせを減らしたい場合。

### 案C: バックエンドに「確定フラグ/手動上書きフラグ」を持たせる
- 有休タスクに `manual_override: true`（や `source: "manual"`）を付与し、同期側はこれを見て確実にスキップ。`task_total_minutes` の正本もこのフラグで判定。
- メリット: 「ユーザー確定済み」を明示でき、将来の同期仕様変更にも頑健。突合・締め処理での判定にも使える。
- デメリット: Firestoreスキーマ追加、既存データのマイグレーション/後方互換の検討が必要。
- 注意: 2.4の現状でも「有休タスクが既にあればスキップ」なので、**最小要件には必須ではない**。堅牢性のための上位互換オプション。

### 案D: 突合（予実）側だけで齟齬を吸収（入力フォームを触らない）
- 入力フォームは変えず、`admin.html` の予実突合表示側で「有休あり日は差異を許容/注記」する。
- メリット: 入力UI（スマホ2面）を触らずに済む。
- デメリット: 「各スタッフが手動訂正する」という要件（1.2）から外れる。ユーザーが直せない。
- 位置づけ: 単独では要件不足。案A/Bの補助として検討。

> **たたき台の推奨**: 「案A をベースに、案B の最小の可視化（バッジ/注記）を載せる。案C はフェーズ2で堅牢化として検討」。最終決定は要確認リスト解消後。

## 6. スコープ分割・MVP・進行フロー

### 6.1 MVPの定義（たたき台）
- **MVP = 工務スマホで「有休あり日に参考値を見ながら時間を訂正し、正しい `task_total_minutes` で確定できる」**。
- 既存の再同期スキップ挙動（2.4）により、訂正値が同期で壊れないことを確認する。

### 6.2 進行順（たたき台・確定は要相談）
1. **工務スマホ**（最小・既存time欄活用でリスク低）→ MVP検証
2. **工務PC**（代理入力に横展開、突合の注記）
3. **ネットスマホ**（時間帯モデルの訂正導線）
4. **ネットPC**（代理入力へ横展開）

> 工務を先に通してパターンを確立 → ネットへ展開、の順。各面は「検知 → 参考値提示 → 上書き確定 → 同期非破壊の確認」を1セットとする。

### 6.3 各面の完了条件テンプレ（案）
- [ ] 有休タスクのある日を検知して表示できる
- [ ] Jobcan参考値（分/時間帯）を併記できる
- [ ] ユーザーが入力値で上書き・確定できる
- [ ] `task_total_minutes` が訂正後の値で正しく保存される
- [ ] 再同期しても訂正値が壊れない
- [ ] 予実突合の表示が訂正後に矛盾しない
